Fix auth token refresh

handy-fox · hotfix · 2m ago
Previous feedback

“The mutex guard approach is correct but the error handling needs to propagate properly to callers.”

Refactored the authentication token refresh flow to eliminate the race condition causing duplicate refresh requests when multiple concurrent API calls each independently detect an expired token.

The core problem: when several requests hit a 401 simultaneously, each spawned its own refresh call. The first would succeed, but subsequent ones would invalidate the token the first just obtained, cascading into a logout.

Changes

  1. Introduced a Mutex<Option<Token>> guard in TokenManager so only one refresh can proceed at a time. Subsequent callers await the lock and receive the already-refreshed token.
  2. Replaced the eager refresh_if_expired() call in the request interceptor with a check-then-lock pattern that reads the token optimistically before acquiring the mutex.
  3. Added a 30-second TTL buffer: tokens are refreshed when within 30s of expiry rather than after they expire, preventing the 401 from triggering in the first place.
  4. Removed the retry-with-refresh logic from ApiClient::send() since the interceptor now guarantees a valid token.
token_manager.rs
async fn refresh_token(&self) -> Result<Token> {
let guard = self.refresh_mutex.lock().await;
if let Some(t) = &*guard {
if !t.expires_within(Duration::from_secs(30)) {
return Ok(t.clone());
}
}
let new_token = self.do_refresh().await?;
*guard = Some(new_token.clone());
Ok(new_token)
}

All 14 existing auth tests pass. Added 3 new concurrent-refresh tests that spawn 50 simultaneous requests against an expiring token. The race condition is no longer reproducible under load.
