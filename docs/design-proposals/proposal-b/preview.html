Authentication Middleware Refactor

This plan proposes extracting JWT validation from the route handlers into a standalone middleware module. The key changes are:

  • Create src/middleware/auth.ts with validateToken() and refreshRotation()
  • Move JwtPayload type to src/types/auth.ts for shared access
  • Add refresh token rotation with 7-day sliding window
  • Implement rate limiting on the /auth/refresh endpoint (10 req/min)
  • Update all protected routes to use the new middleware instead of inline validation

Testing Strategy

Unit tests for token validation logic, integration tests for the refresh flow, and update existing route tests to work with middleware-based auth.

Description

Extract the JWT validation logic from individual route handlers into a reusable authentication middleware. The current implementation duplicates token parsing in 12+ route files. We need a centralized middleware that handles token validation, refresh rotation, and error responses.

Which database migration approach should I use for the schema changes?
The project currently has no migration framework configured. I found both Diesel and Refinery as dependencies in adjacent crates, so either would be consistent with the codebase.
Refinery migrations
SQL-based migrations with version numbering. Matches the existing pattern in orkestra-core.
Raw SQL scripts
Manual SQL files without a framework. Simpler but no automatic version tracking.
Diesel migrations
Full ORM-based migrations. More powerful but adds a heavier dependency.
