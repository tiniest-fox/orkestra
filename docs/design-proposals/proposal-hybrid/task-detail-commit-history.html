Design Proposal · Feed UI · Diff Tab

Task Drawer — Commit History

The Diff tab gains a Branch / Commits mode toggle at the top of its content area. Branch mode is the existing cumulative diff view (unchanged). Commits mode replaces the file list sidebar with a commit list — same 224px width — and shows the per-commit diff on the right. The toggle is a secondary control, lighter than the tab bar, with no colored underline indicator. Four states are documented: Branch mode (default), Commits with a commit selected, Commits with the diff loading, and Commits with an empty branch.

State 1 — Diff tab · Branch mode (default)
The Diff tab's default view is unchanged — cumulative diff of the whole branch. The mode toggle is now visible at the top of the content area, with "Branch" pill active. It reads as a subordinate control, not competing with the tab bar above it.

The mode toggle lives inside the Diff tab content area, between the tab bar and the diff itself. It is not part of the DrawerTabBar component and does not change the tab bar height or layout.
Toggle styling: two pill buttons inside a single rounded container, canvas background, border border. Active pill gets surface-2 fill. No colored underline — intentionally lighter than the tab bar to signal it is a secondary control.
The g hotkey toggles between Branch and Commits within the Diff tab. It does not conflict with any existing hotkey registered in DrawerDiffTab (c, ], [, j, k).
Branch mode is the default. The mode selection resets to Branch when the drawer closes, matching the lifecycle of tab selection.
State 2 — Diff tab · Commits mode · Commit selected
Commits mode replaces the file list sidebar with a commit list (same 224px width). The most recent commit is auto-selected. The right pane shows that commit's diff, with a slim header showing the sha and subject line. Navigating the list with j/k or click updates the right pane.

The diff pane header renders immediately when a commit is selected — sha and subject are known from the commit list data. Only the file diff content needs to load from the backend.
The commit list remains interactive during loading. If the user navigates to a third commit before the second finishes loading, the second's request is abandoned and the third's begins. This is standard cancellation behaviour from the useChunkedHtml hook.
The skeleton uses the same surface-3 shimmer pattern as DiffSkeleton in the existing codebase — no new animation is introduced.
There is no explicit "loading" label or spinner. The skeleton communicates the loading state without words. This matches the Branch mode behaviour where the skeleton appears without a status message.
State 4 — Diff tab · Commits mode · No commits yet
The task's branch has no commits. This is valid early in a task's life — the agent may have been spawned but not yet completed any work. The toggle is still visible. The split layout collapses to a full-width empty state in the commit list pane; the diff pane is not shown.

Empty state uses the standard mono / text-3 pattern used throughout the drawer for all empty states (no artifact, no logs, no subtasks yet). No illustration, no call to action — there is nothing the user can do here.
When no commits exist, the split layout is suppressed. There is nothing to show in the list and no diff to render in the right pane. A full-width message is the correct treatment.
The count "0 commits" next to the toggle reinforces the empty state without the user needing to look at the content area. The j k nav hint is omitted from the toggle bar when count is 0.
Branch mode shows "No changes." in this same task state (if no uncommitted work exists yet). Both modes gracefully handle the early-in-task case.
