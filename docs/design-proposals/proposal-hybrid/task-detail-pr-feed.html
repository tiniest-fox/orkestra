Task Drawer — PR Detail

When a done task has a pr_url, the drawer shows a dedicated PR tab as its default view. Four states are documented below: happy path (checks pass, approved), failing checks, changes requested with inline review comments, and merge conflicts. Each state has a distinct call to action and a consistent information hierarchy: PR state first, CI second, review third, action last.

State 1 — Open · Checks passing · Approved
The PR is ready to merge. CI is green. One reviewer approved. Primary action is Merge. "View PR" is secondary — the user may want to read the diff on GitHub before merging.

When checks are failing, Merge is not offered. The only footer action is "View on GitHub" which links directly to the PR where the user can inspect failure logs.
Failing check rows use a red tint background (var(--red-bg)) — visually distinct but not alarming. The check detail field shows the failure reason in muted red.
There is no "retry checks" or "address failures" action here. The intended flow is: user reads the failures, understands what broke, then manually triggers an "address PR comments" workflow if they choose — or uses reject/feedback to send the agent back to work on the failing tests.
The status line at the bottom summarizes the failure count so the user never has to open the tab to know the state.
State 3 — Open · Changes requested · Comments selected
A reviewer has requested changes and left inline comments on specific files. The user selects which comments to address, optionally adds guidance, then clicks "Address Comments" to send the agent back to work. This is the core "address PR comments" workflow from PrIssuesPanel — redesigned for the drawer context.

Comments default to unchecked. The user selects which ones to address. This is intentional — not all review comments require agent action (some may be nits the user will handle manually).
The selection summary bar appears as soon as any comment is checked. It shows "N of M comments selected" and a clear button. It lives between the section header and the review list — not in the footer — so the user can see it while scrolling through comments.
Unchecked comments fade to 55% opacity. They remain visible (so the user remembers what they decided to skip) but visually step back. This is preferable to hiding them.
"Address N comments" as the button label is more precise than "Address Comments" — it confirms the selection count and reduces the risk of clicking before selecting.
The guidance textarea is shown inline below the comments. It is labeled "optional" and has no required indicator. The placeholder sets expectations: it is for context, not instructions.
When comments are selected, Merge is not offered. The task will transition back to a work stage when "Address N comments" is clicked — it is no longer in a mergeable state.
State 4 — Open · Merge conflicts
The PR branch has diverged from the base branch and has conflicts. No comment workflow is available until conflicts are resolved. "Fix Conflicts" sends the agent back to work to rebase and resolve. Reviews and comments are shown in a suppressed state with an explanatory note.

The PR state badge changes to amber "conflicts" when merge_state_status === "DIRTY" or mergeable === false. The status bar also adopts amber tinting. This is distinct from the open/green state used when checks pass.
Reviews are shown at reduced opacity with a note explaining they cannot be addressed yet. They are not hidden — the user needs to know what's coming after conflicts are resolved so they can mentally prepare the work. But they are non-interactive (pointer-events: none).
"Fix Conflicts" (amber) is the sole primary action. There is no "Address Comments" button — the system enforces the ordering: resolve conflicts first, then comments. This matches the logic in PrIssuesPanel.
The conflict panel explains what the agent will do (rebase, resolve, re-run checks) rather than just saying "conflicts exist." Users who aren't familiar with git conflict resolution need this context to trust the automated path.
The status line summarizes "merge conflicts" so the user knows the state at a glance from the feed without opening the drawer.
Implementation Notes

Changes needed in TaskDrawer

availableTabs()
When is_done and pr_url is set, replace the current ["diff", "artifact", "history"] tab set with ["pr", "diff", "artifact", "history"]. The PR tab uses hotkey p.
defaultTab()
When is_done and pr_url is set, default tab is "pr". Without a PR URL the existing "diff" default remains.
DrawerTabId type
Add "pr" to the DrawerTabId union type and add a corresponding case in the tab body switch.
Footer — done state
The done-state footer stays as-is when no PR URL is set. When a PR URL is set, footer changes based on PR state: conflicts → Fix Conflicts (amber); comments selected → Address N comments (peach); clean → Merge (peach) + View PR.
New file: DrawerPrTab.tsx
Extract the PR tab content into DrawerPrTab.tsx alongside the existing drawer files. It takes prUrl, taskId, baseBranch and exposes onConflictsDetected / onCommentsSelected callbacks so the footer can render the right action.
PrIssuesPanel fate
The logic from PrIssuesPanel.tsx moves into DrawerPrTab.tsx. The workflow_address_pr_conflicts and workflow_address_pr_comments Tauri invocations remain identical. No backend changes required.
